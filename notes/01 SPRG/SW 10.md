# IT Security: Input Validation & Output Encoding

## 1. Injections: Das Konzept
Injections treten auf, wenn nicht vertrauenswürdige Benutzereingaben ohne ausreichende Prüfung an einen Interpreter weitergegeben werden.
* **Ablauf**: Ein Parser oder Interpreter führt bösartigen Code aus, der in den Daten versteckt war.
* **Vulnerable Schnittstellen**: Viele verschiedene Schnittstellen können betroffen sein, sobald Daten die Vertrauensgrenze überschreiten.

---

## 2. SQL Injection (SQLi)
Eine der gefährlichsten Injection-Arten, bei der Datenbankabfragen manipuliert werden.
* **Arten von SQLi**:
    * **In-band**: Ergebnisse werden über denselben Kanal zurückgegeben (z. B. UNION-basiert).
    * **Inferential (Blind)**: Informationen werden über Zeitverzögerungen (Sleep) oder Fehlermeldungen rekonstruiert.
    * **Out-of-band**: Die Ergebnisse werden über einen anderen Kanal (z. B. E-Mail) gesendet.
* **Techniken**: Manipulation der WHERE-Klausel oder Auslesen fremder Tabellen mit UNION-Queries.

---

## 3. Cross-Site Scripting (XSS)
Einschleusen von bösartigem JavaScript, das im Browser des Opfers ausgeführt wird.
* **Stored XSS**: Der bösartige Code wird dauerhaft auf dem Server gespeichert (z. B. in Foren oder Kommentaren).
* **Reflected XSS**: Der Code wird durch eine Benutzereingabe (z. B. in der URL) direkt reflektiert.
* **DOM-based XSS**: Die Schwachstelle liegt in der clientseitigen Verarbeitung von Daten im Document Object Model.

---

## 4. Cross-Site Request Forgery (CSRF)
Ein Angriff, bei dem der Browser des Opfers gezwungen wird, eine ungewollte HTTP-Anfrage an eine Seite zu senden, bei der der Nutzer eingeloggt ist.
* **Beispiel**: Eine bösartige Website löst eine Überweisung auf einer im Hintergrund geöffneten Bank-Seite aus.

---

## 5. Präventionsmaßnahmen
* **Parametrisierte APIs**: Nutzung von Prepared Statements zur Trennung von Code und Daten.
* **Input Validation**: Prüfung auf Datentyp, Länge und Syntax (Whitelisting ist am sichersten).
* **Contextual Output Encoding**: Daten müssen vor der Anzeige je nach Kontext (HTML, JS, CSS) maskiert werden.
* **CSP (Content Security Policy)**: Steuerung durch den Browser, welche Ressourcen geladen werden dürfen.
* **Anti-CSRF Tokens**: Einzigartige Token für jede Sitzung zur Verifizierung von Anfragen.

---

## Wichtigste Key Takeaways
- [ ] **Eingabeprüfung ist Pflicht**: Jede Eingabe von außen muss als potenziell bösartig eingestuft werden.
- [ ] **Whitelisting**: Erlaube nur das, was explizit bekannt und sicher ist.
- **Prepared Statements**: Verhindern SQL-Injections zuverlässig durch Trennung von Befehl und Wert.
- [ ] **Encoding statt Löschen**: Maskiere Sonderzeichen bei der Ausgabe, um XSS zu verhindern.
- [ ] **Sicherheits-Header nutzen**: CSP und SameSite-Cookie-Attribute bieten zusätzliche Schutzebenen.

# IT Security: Authentication & Session Management

## 1. Kryptographische Grundlagen
Sichere Identitätsprüfung basiert auf kryptographischen Kernfunktionen:
* **Symmetrisch/Asymmetrisch**: Verschlüsselung mit geteilten oder öffentlichen/privaten Schlüsseln.
* **Hashing**: Erzeugung von digitalen Fingerabdrücken für Passwörter.
* **TLS**: Verschlüsselung der Transportebene zur Verhinderung von Man-in-the-Middle (MITM) Angriffen.

---

## 2. Authentifizierung vs. Autorisierung
* **Authentifizierung**: "Wer bist du?" (Identitätsnachweis).
* **Autorisierung**: "Was darfst du?" (Rechteprüfung).

---

## 3. Sichere Authentifizierung
* **Passwort-Speicherung**: Passwörter dürfen niemals im Klartext gespeichert werden, sondern nur als sichere Hashes.
* **Fehlermeldungen**: Nutze generische Meldungen wie "Benutzername oder Passwort falsch", um Enumeration zu verhindern.
* **MFA (Multi-Faktor)**: Kombination aus Wissen (Passwort), Besitz (Token) oder Inhärenz (Biometrie).
* **Schutz gegen Angriffe**: Implementierung von Rate-Limiting und temporären Sperren bei Fehlversuchen.

---

## 4. Session Management
Sitzungen müssen nach dem Login sicher verwaltet werden, um Übernahmen zu verhindern.
* **Session Hijacking**: Diebstahl einer gültigen Sitzungs-ID.
* **Session Fixation**: Unterschieben einer dem Angreifer bekannten Sitzungs-ID.
* **Schutzmaßnahmen**:
    * Generierung langer, zufälliger IDs auf der Serverseite.
    * Erneuerung der ID nach jedem erfolgreichen Login.
    * Setzen von sicheren Cookie-Attributen (Secure, HttpOnly).
    * Implementierung von Timeouts und korrektem Logout.

---

## 5. Token-basierte Authentifizierung (JWT)
JSON Web Tokens ermöglichen zustandslose (stateless) Authentifizierung.
* **Struktur**: Header, Payload (Nutzerdaten) und Signature.
* **Wichtige Regeln**:
    * Validierung der Signatur bei jedem Request.
    * Algorithmus "None" darf niemals akzeptiert werden.
    * Keine sensiblen Daten im unverschlüsselten Payload speichern.

---

## Wichtigste Key Takeaways
- [ ] **Multi-Faktor-Authentifizierung**: Ist die effektivste Methode gegen gestohlene Zugangsdaten.
- [ ] **Passwort-Hygiene**: Nutze starke Hashing-Algorithmen und erzwinge Passwort-Policys.
- [ ] **Session IDs schützen**: Sitzungs-IDs müssen wie Passwörter behandelt und nach dem Login rotiert werden.
- [ ] **TLS überall**: Ohne verschlüsselte Verbindung sind alle anderen Sicherheitsmaßnahmen hinfällig.
- [ ] **Keine Eigenbau-Krypto**: Nutze immer etablierte Bibliotheken für Verschlüsselung und JWT-Handling.
