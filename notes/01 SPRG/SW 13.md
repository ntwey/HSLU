## 2. Historischer Kontext & Malware
* **Internet-Design:** Sicherheit war bei der Entwicklung des Internets (vom ARPANET zum World Wide Web) kein primäres Kriterium. Der Fokus lag auf dem weltweiten Datenaustausch.
* **Herausforderungen heute:** Anonymität, Datenlecks und Cyberkriminelle.
* **Malware-Arten:** Ransomware, Spyware, Adware, Trojaner, Würmer, Viren und mehr.
* **Der Morris-Wurm (1988):** Der erste Internet-Wurm, der entwickelt wurde, um die Anzahl der Maschinen im Netz zu zählen. Er nutzte unter anderem ein Passwort-Wörterbuch und einen **Buffer Overflow**, um sich selbst zu replizieren.
## 3. Technische Funktionsweise eines Buffer Overflows
Ein Buffer Overflow entsteht, wenn eine Anwendung Daten in einen definierten Speicherbereich (Buffer) schreibt, die zugewiesene Größe des Buffers jedoch nicht überprüft wird. Dadurch werden benachbarte Speicherbereiche überschrieben.
```c
#include <stdio.h>
#include <string.h>
int main() {
    char buffer[4]; // Reserviert nur 4 Bytes auf dem Stack
    printf("Enter your input: ");
    gets(buffer);   // SEHR UNSICHER: liest unbegrenzt Input ein
    printf("You entered: %s\n", buffer);
    return 0;
}
```
* **Auswirkung bei zu langem Input:** Wenn der Benutzer eine lange Zeichenkette (z. B. `abcdefghijklmnopqrstuvwxyz`) eingibt, läuft der Speicherbereich über. Dies führt zu einem kritischen Systemfehler: **Segmentation fault (core dumped)**.

### Der Stack-Mechanismus:
* Ein Stack-Frame (z. B. für eine Funktion `echo`) speichert lokale Variablen (wie den Puffer `buf`) und direkt dahinter die **Return Address** (Rücksprungadresse zur aufrufenden Funktion, z. B. `main`).
* **Wachstumsrichtung:** Wenn Daten in den Puffer geschrieben werden, bewegen sie sich bei einem Überlauf in Richtung der höheren Speicheradresse (`%rsp + 0x18`) – genau dorthin, wo die Return Address liegt.
* **Exploitation:** Ein Angreifer füllt den Puffer gezielt mit Daten (Padding), überschreibt die legitime Return Address mit einer manipulierten Adresse und biegt den Kontrollfluss des Programms auf eigenen Schadcode (**Exploit Code**) ab, der ebenfalls im Stack liegen kann.
## 4. Schutzmechanismen & Workarounds

### Eingebaute Betriebssystem- und Compiler-Protections:
1. **Stack Canaries (Stack-Sicherheitswerte):** Vom Compiler eingefügte Prüfungen, die einen Schutzwert vor der Return Address platzieren, um Manipulationen zu erkennen.
2. **ASLR (Address Space Layout Randomisation):** Randomisiert die Speicheradressen von Programmkomponenten bei jedem Start, sodass Angreifer Sprungziele nicht vorhersagen können.
3. **DEP / NX (Data Execution Prevention / Non-executable code segments):** Markiert Speicherbereiche (wie den Stack) explizit als "nicht ausführbar". Alles, was beschreibbar ist, darf von der CPU nicht als Code ausgeführt werden.

### Umgehung von DEP: Return-Oriented Programming (ROP)
* Da durch DEP kein eigener Code auf dem Stack ausgeführt werden darf, nutzen Angreifer **ROP**.
* Herebei wird kein neuer Code eingeschleust. Stattdessen sucht der Angreifer nach bereits vorhandenen, legitimen Code-Fragmenten im ausführbaren Speicher, die auf eine `ret` (Return)-Anweisung (`c3`) enden (sogenannte **Gadgets**).
* Durch das gezielte Manipulieren von Return-Adressen auf dem Stack werden diese Gadgets zu einer bösartigen Kette verknüpft. Da der Code selbst legitim und im erlaubten Speicher liegt, greift DEP hierbei nicht.

### Gegenmaßnahmen für Entwickler (Mitigation):
* **Sichere Sprachen nutzen:** Wenn möglich, Programmiersprachen verwenden, die das Speichermanagement automatisch im Hintergrund handhaben.
* **Input-Längen validieren:** Vor dem Schreiben in einen Buffer immer strikt die Länge des Inputs prüfen.
* **Sichere Funktionen verwenden:** Niemals unsichere Funktionen wie `gets()` oder `strcpy()` nutzen.
* **Compiler-Optionen aktivieren:** Moderne Sicherheitsfeatures wie Stack Canaries, DEP/NX-Support und ASLR beim Kompilieren standardmäßig erzwingen.
## 5. Übungs- und Beispielfragen aus der Vorlesung

### LFI (Local File Inclusion) Beispiel
* **Code**:
```php
	  <?php
$page=$_GET['page'];
include($page . '.php');
?>
```

* **Schwachstelle:** Lokale Datei-Inklusion (LFI) / Pfad-Traversierung. Der Benutzer kontrolliert unvalidiert über einen GET-Parameter, welche Datei vom Server geladen und ausgeführt wird.
* **Verhinderung (Mitigation):** Eingaben gegen eine Whitelist erlaubter Seiten prüfen, Pfadkomponenten säubern (Sanitization) oder ID-basiertes Mapping statt direkter Strings verwenden.

### CSRF (Cross-Site Request Forgery) Bedingungen
* Damit ein CSRF-Angriff potenziell möglich ist, müssen bestimmte Voraussetzungen gegeben sein:
  - Authentisierungs- oder Session-Informationen (wie Cookies) werden vom Browser **mit jedem Request automatisch mitgeschickt**.
  - Das Session-Cookie darf **nicht** restriktiv geschützt sein (ein Attribut wie `SameSite=Strict` würde CSRF im Cross-Site-Kontext verhindern).

### DOM-based XSS (Cross-Site Scripting)
* **Zutreffende Aussagen**:
  - Die Payload wird in der Regel **nicht zum Server geschickt**.
  - Das Sicherheitsrisiko spielt sich **immer auf der Client-Seite** ab (Verarbeitung direkt im Browser über das Document Object Model).

## Key Takeaways
1. **Niemals gets() oder unbeschränkte Funktionen verwenden:** Die manuelle Speicherverwaltung in Sprachen wie C/C++ ist die Wurzel von Buffer Overflows. Eine fehlende Längenprüfung führt unweigerlich zu Systemabstürzen oder Code-Execution.
2. **Sicherheit ist ein Wettrüsten (ROP vs. DEP):** Auf jede neue Betriebssystem-Härtung (wie DEP/NX) finden Angreifer neue logische Umgehungen (wie ROP unter Nutzung vorhandener "Gadgets"). Echte Sicherheit erfordert daher immer mehrere Schutzschichten ("Defense in Depth").
3. **Das Grundproblem ist mangelndes Vertrauen in Input:** Ob Buffer Overflow in C, Local File Inclusion (LFI) in PHP oder DOM XSS im Browser-JavaScript – das fundamentale Problem ist fast immer das unvalidierte Vertrauen in benutzergesteuerte Eingabedaten.
