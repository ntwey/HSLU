# IT Security: Authorisation (Zusammenfassung)

## 1. Grundlagen der Autorisierung
Die Autorisierung befasst sich mit der Verwaltung von Berechtigungen für **Subjekte**, die auf **Objekte** zugreifen.
* **Formel:** `permission = access(subject, object)`
* **Zugriffsarten:**
    * Create (Erstellen)
    * Read (Lesen)
    * Update (Aktualisieren)
    * Delete (Löschen)
    * Execute (Ausführen)

## 2. Methoden und Modelle
Es wird zwischen verschiedenen Ansätzen zur Zugriffskontrolle unterschieden:

### Methoden & Policies
* **DAC (Discretionary Access Control):** Zugriff liegt im Ermessen des Eigentümers.
* **MAC (Mandatory Access Control):** Systemweite, zwingende Vorgaben.
* **Open Policy vs. Closed Policy:** Grundsätzliches Erlauben oder Verweigern von Zugriffen.

### Modelle
* **RBAC:** Role-Based Access Control (Rollenbasiert)
* **ABAC:** Attribute-Based Access Control (Attributbasiert)
* **PBAC:** Policy-Based Access Control (Richtlinienbasiert)

## 3. Architektur & API-Struktur
Die Umsetzung erfolgt oft nach dem **XACML-Modell**:
* **PEP (Policy Enforcement Point):** Erzwingt die Entscheidung.
* **PDP (Policy Decision Point):** Trifft die Entscheidung basierend auf Regeln.
* **PAP (Policy Administration Point):** Verwaltung der Richtlinien.
* **PIP (Policy Information Point):** Liefert zusätzliche Attribute für die Entscheidung.

### API-Typen
* **Ressourcenorientiert (REST):** Business Logic oft im Frontend, Zugriffskontrolle auf Objektebene (Object-level), HTTP-Methoden (GET, POST, etc.) bestimmen die Aktion.
* **Serviceorientiert (SOA):** Business Logic im Backend, Zugriffskontrolle auf Funktionsebene (Function-level).

## 4. Sicherheitsrisiken (Broken Access Control)
Häufige Schwachstellen sind:
* **IDOR (Insecure Direct Object Reference):** Direkter Zugriff auf Datensätze über IDs (z. B. `?id=1234`) ohne Prüfung der Eigentümerschaft.
* **Path Traversal:** Umgehen von Filtern durch Navigieren in Verzeichnissen (z. B. `../../etc/passwd`).
* **Client-Side Caching:** Sensible Daten bleiben im Browser-Cache.
* **Fehlende serverseitige Prüfung:** Zugriffskontrolle darf niemals rein im Frontend stattfinden.

## 5. Best Practices
* **Closed Policy:** Alles verbieten, was nicht explizit erlaubt ist.
* **Least Privilege:** Nur so viele Rechte wie unbedingt nötig vergeben.
* **Serverseitige Durchsetzung:** Jede Anfrage muss im Backend geprüft werden.
* **Unvorhersehbare IDs:** Nutzung von UUIDs statt fortlaufender Nummern.
* **Token-Management:** JWT-Tokens signieren und nach dem Logout invalidieren.
* **Segregation of Duties:** Trennung von Aufgaben (z. B. Entwickler vs. Produktion).

## 6. OAuth 2.0 & OIDC (OpenID Connect)
* **OAuth:** Fokus auf **Autorisierung** (Delegierung von Zugriffen via Tokens).
* **OIDC:** Fokus auf **Authentifizierung** (Identitätsprüfung, "Login mit Social Media").
* **Wichtige Flows:**
    * Authorisation Code Flow (Sicherster Standard).
    * PKCE (Proof Key for Code Exchange) zur Absicherung von Apps.
    * *Veraltet:* Implicit Flow und Password Flow.

---

## Wichtigste Key Take-Aways
* **Backend is King:** Verlasse dich niemals auf Frontend-Logik; die Autorisierung muss bei jedem Request serverseitig validiert werden.
* **Deny by Default:** Implementiere immer eine "Closed Policy", um Sicherheitslücken durch vergessene Berechtigungsregeln zu minimieren.
* **Identität vs. Erlaubnis:** Authentifizierung (wer bist du?) ist nicht gleich Autorisierung (was darfst du?).
* **Schutz vor IDOR:** Überprüfe bei jedem Zugriff auf ein Objekt (z. B. ein Bankkonto), ob das Subjekt tatsächlich der Besitzer dieses spezifischen Objekts ist.
* **Moderne Standards:** Nutze für APIs den Authorisation Code Flow mit PKCE statt veralteter OAuth-Methoden.
