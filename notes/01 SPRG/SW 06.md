# Zusammenfassung: UML Use Case Diagramme & Cybersecurity

## 1. Grundlagen von Use Case Diagrammen (UCD)
Use Case Diagramme werden während der Systemanalyse genutzt, um **funktionale Anforderungen** aus der Perspektive des Endnutzers zu modellieren. Sie beschreiben das "Was" (Interaktion) und den "Zweck" (Ziel), ignorieren jedoch Implementierungsdetails.

### Die 4 Hauptelemente
* **Akteure (Actors):** Personen oder externe Systeme/Geräte, die mit dem System interagieren.
* **Systemgrenze (Boundary):** Definiert den Rahmen des betrachteten Systems.
* **Use Cases:** Die Dienste oder Prozesse, die das System ausführt (dargestellt als Elipsen).
* **Beziehungen:** Linien, die die Interaktion zwischen Akteuren und Use Cases aufzeigen.

---

## 2. Fortgeschrittene Modellierungstechniken
* **Generalisierung/Spezialisierung:** Akteure (z. B. User -> Administrator) und Use Cases (z. B. Payment -> Credit Card) können hierarchisch strukturiert werden.
* **Beziehungen zwischen Use Cases:**
    * `<<include>>`: Zeigt an, dass ein Use Case zwingend einen anderen benötigt (z. B. "Check-In" inkludiert "Weigh Luggage").
    * `<<extend>>`: Beschreibt optionales Verhalten unter bestimmten Bedingungen.
* **Kontextmodell:** Eine vereinfachte Darstellung des gesamten Systems als ein einziger Use Case, um alle externen Schnittstellen auf einen Blick zu zeigen.

---

## 3. Security Engineering mit Use Cases
Ein zentraler Aspekt der Präsentation ist die Erweiterung klassischer Diagramme um Sicherheitsbetrachtungen:

### Misuse & Mitigation Cases
* **Misuse Cases:** Beschreiben böswillige Handlungen eines **Angreifers** (z. B. Injection Attack, Identity Theft), die das System bedrohen.
* **Mitigation Cases:** Funktionen, die eingeführt werden, um die Bedrohung durch einen Misuse Case zu mindern (z. B. Input Validation, Verschlüsselung).

---

## 4. Schnittstellen und Eingabevalidierung
Die größte Schwachstelle in Systemen sind oft **Schnittstellen**. 
* **Das Problem:** Wenn Sender und Empfänger Daten unterschiedlich interpretieren ($Dec(Enc(M)) \neq M$), entstehen Sicherheitslücken.
* **Ziel des Angreifers:** Kombinationen aus Daten und Protokollen zu finden, bei denen der Empfänger die Daten anders verarbeitet als vom Absender (oder Entwickler) vorgesehen.
* **Lösung:** Strenge **Input Validation**. Nur valide Daten dürfen akzeptiert werden; die "Sprache" an der Schnittstelle muss auf beiden Seiten identisch interpretiert werden.

---

## Wichtigste Key Take-Aways
* **Perspektive:** Use Cases dokumentieren das Systemverhalten aus Nutzersicht, nicht die technische Umsetzung.
* **Sicherheitsfokus:** Cybersecurity beginnt bereits in der Analysephase durch das Modellieren von Fehlbedienungen (**Misuse Cases**).
* **Validierung:** "Secure Input Validation" ist die wichtigste Verteidigung gegen typische Exploits wie SQL-Injection oder Buffer Overflows.
* **Schnittstellen-Design:** Design-Contracts müssen explizit definiert und nachweisbar befolgt werden, um Interpretationsfehler zu vermeiden.
