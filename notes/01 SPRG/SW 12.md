## 1. Was ist OWASP?
Das **Open Worldwide Application Security Project (OWASP)** ist eine weltweite, gemeinnützige Organisation, die sich der Verbesserung der Softwaresicherheit verschreibt.

### Wichtige OWASP-Projekte und Initiativen:
* **Top 10:** Liste der kritischsten Sicherheitsrisiken für Webanwendungen.
* **Cheat Sheet Series:** Praktische Leitfäden für Entwickler.
* **ZAP (Zed Attack Proxy):** Ein Tool zum Finden von Sicherheitslücken.
* **Weitere:** Dependency Track, OWASP SAMM, SecurityShepherd, Security Knowledge Framework (SKF), ModSecurity Core Rule Set.

---

## 2. Die OWASP Top 10 im Wandel
Die Top 10 wird regelmäßig aktualisiert, um der sich verändernden Bedrohungslandschaft gerecht zu werden. 

### Wichtige Trends & Verschiebungen:
* **Broken Access Control (Fehlerhafte Zugriffskontrolle):** Rückte von Platz 5 (2017) auf Platz 1 (2021/2025) vor und bleibt das kritischste Risiko.
* **Insecure Design (Unsicheres Design):** Kam 2021 neu hinzu (A04) und betont, dass Sicherheit bereits in der Architekturphase bedacht werden muss.
* **Software Supply Chain Failures:** Ein neuer, hochrelevanter Fokus in der aktuellen Bedrohungslandschaft (A03:2025).

## 3. Fokus-Vulnerabilities aus den Web Top 10

### A02: Cryptographic Failures (Kryptografische Fehler)
* **Probleme:** Ungenügende Verschlüsselung von Daten bei der Übertragung (in transit) oder Speicherung (at rest), Nutzung schwacher Algorithmen sowie ungesalzene oder im Klartext gespeicherte Passwörter.
* **Betroffene Daten:** Passwörter, Kreditkarteninformationen, Gesundheitsdaten, PII (Personenbezogene Daten).
* **Gegenmaßnahmen:** - Keine sensiblen Daten unnötig speichern.
  - Starke Algorithmen und sichere Zufallsgeneratoren nutzen.
  - Sicheres Schlüsselmanagement (Key Management) etablieren.
  - Passwörter mit sicheren, gesalzenen Hashes speichern.
  - **HSTS-Header** setzen (HTTP Strict Transport Security).

### A05: Security Misconfiguration (Sicherheitskonfiguration)
* **Probleme:** Fehlende Systemhärtung, ungenutzte aktivierte Features, Standard-Accounts (Default Logins) und zu detaillierte Fehlermeldungen, die Angreifern Hinweise liefern.
* **Gegenmaßnahmen:** Wiederholbare Härtungs- und Deployment-Prozesse, minimale Plattformkomponenten verwenden, Konfigurationen regelmäßig prüfen und eine segmentierte Anwendungsarchitektur aufbauen.

#### Spezialfall: XML External Entities (XXE)
* **Konzept:** Angriffe auf XML-Parser über DTDs (Document Type Definitions), bei denen eigene oder externe Entities definiert werden (z. B. `<!ENTITY xxe SYSTEM "file:///etc/shadow" >`).
* **Auswirkungen:** Datenabfluss (Data Leak), Denial of Service (DoS), Remote Code Execution (RCE) oder Server-Side Request Forgery (SSRF).
* **Gegenmaßnahmen:** Nutzung anderer Datenformate wie JSON, Deaktivierung der Verarbeitung von externen Entities/DTDs im XML-Parser, Validierung von XML/XSL.

### A06: Vulnerable and Outdated Components
* **Probleme:** Veraltete oder nicht mehr unterstützte Software-Komponenten, unzureichendes Patchmanagement und fehlende Tests.
* **Gegenmaßnahmen:** Unnötige Abhängigkeiten entfernen, regelmäßige und risikobasierte Updates durchführen, Komponenten kontinuierlich überwachen und nur aus offiziellen Quellen beziehen.

### A08: Software and Data Integrity Failures
* **Probleme:** Unverifizierte Softwarepakete, Einspielen unvertrauenswürdiger Daten, unsichere CI/CD-Pipelines.
* **Gegenmaßnahmen:** Signaturen von Daten/Software prüfen, vertrauenswürdige Repositories nutzen, Review-Prozesse etablieren, CI/CD-Pipelines absichern.

#### Spezialfall: Deserialisierung
* **Konzept:** Umwandlung eines Bytestreams zurück in ein Objekt (oft genutzt für Persistenz in Dateien, Datenbanken oder im Speicher). Wenn die Quelle unvertrauenswürdig ist, kann dies zu RCE führen.
* **Gegenmaßnahmen:** Deserialisierung unvertrauenswürdiger Daten vermeiden, Integritätsprüfungen vorschalten, Typ-Einschränkungen (Type Constraints) erzwingen, Prozesse isolieren.

### A09: Security Logging and Monitoring Failures
* **Probleme:** Unzureichende Logs, fehlendes Echtzeit-Monitoring/Alerting, rein lokale Log-Speicherung.
* **Gegenmaßnahmen:** Audit-Trails einrichten, Logs zentralisieren und normalisieren, effektive Alarmierung aufbauen, Incident-Response-Pläne bereithalten (Schnittstelle: OWASP AppSensor Projekt).

### A10: Server-Side Request Forgery (SSRF)
* **Konzept:** Ein Angreifer missbraucht eine Webanwendung, um Anfragen an interne Systeme (hinter der Firewall) oder Drittanbietersysteme zu senden.
* **Gegenmaßnahmen:** Keine direkt vom Benutzer manipulierbaren Inputs in URLs erlauben, Whitelist-basierte Validierung implementieren, Netzwerksegmentierung durchführen.

## 4. OWASP API Security Top 10
Fokussiert sich auf die speziellen Risiken moderner, API-basierter Anwendungen.

### Wichtige API-Schwachstellen im Detail:

#### API3: Broken Object Property Level Authorization (inkl. Mass Assignment)
* **Problem:** APIs erlauben es manchmal, schützenswerte Objekteigenschaften direkt auszulesen oder zu verändern.
* **Mass Assignment:** Ein Endpunkt bindet Client-Eingaben direkt an interne Datenmodelle. Ein Angreifer fügt zusätzliche Parameter hinzu (z. B. `&isAdmin=true` oder `"balance": 500`), um Daten zu manipulieren, die er nicht ändern dürfte.
* **Gegenmaßnahmen:** Zugriffrechte explizit prüfen, direktes Binding vermeiden (Data Transfer Objects nutzen), zurückgegebene Daten minimieren, Antworten validieren.

#### API4: Unrestricted Resource Consumption
* **Problem:** Keine Limits für API-Anfragen, was zu DoS oder massiven Cloud-Infrastrukturkosten führt.
* **Gegenmaßnahmen:** Ratenbegrenzungen (Rate Limits) definieren, maximale Datengrößen erzwingen, Ausgaben-Limits und Alerts einrichten.

#### API6: Unrestricted Access to Sensitive Business Flows
* **Problem:** Automatisierter Missbrauch legitimer Geschäftsabläufe (z. B. Ticket-Scalping durch Bots).
* **Gegenmaßnahmen:** Device-Fingerprinting, Erkennung von menschlichem vs. nicht-menschlichem Verhalten (Captchas etc.), Blockieren bekannter Proxies.

#### API9: Improper Inventory Management
* **Problem:** Veraltete, unvollständige Dokumentation; vergessene Test- oder Alt-APIs (Shadow APIs) sind im Netz exponiert.
* **Gegenmaßnahmen:** Alle API-Hosts inventarisieren, automatisierte Dokumentation nutzen, alte API-Versionen in das Patch-Management einbeziehen.

#### API10: Unsafe Consumption of APIs
* **Problem:** Blindes Vertrauen in APIs von Drittanbietern, was zu SQL-Injections, Datenabfluss oder Open Redirects führen kann.
* **Gegenmaßnahmen:** Sicherheitsüberprüfungen von Drittanbieter-APIs, Verschlüsselung erzwingen, alle empfangenen Daten strikt validieren.

---

## Key Takeaways
1. **Broken Access Control ist die Nummer 1:** Die Absicherung, wer auf welche Daten zugreifen darf, ist historisch und aktuell das größte Problem bei Webanwendungen.
2. **Shift Left / Insecure Design:** Sicherheit muss von Anfang an (Architekturphase) eingeplant werden; reines nachträgliches "Flicken" von Code reicht nicht aus.
3. **API-Sicherheit erfordert andere Muster:** Bei APIs sind vor allem Mass-Assignment-Schwachstellen (automatisches Binding) und unbegrenzter Ressourcenverbrauch (API-DoS) kritische Einfallstore.
