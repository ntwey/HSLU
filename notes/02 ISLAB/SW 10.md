## 1. Theoretische Grundlagen

### Container vs. Virtuelle Maschinen (VMs)
* **Speicherplatz & Ressourcen:** Docker-Container benötigen signifikant weniger Speicherplatz und RAM als VMs, da sie kein vollständiges Gast-Betriebssystem (Guest OS) enthalten, sondern den Kernel des Host-Systems teilen.
* **Steuerung:** Virtuelle Maschinen werden über einen **Hypervisor** gesteuert, während Docker-Container direkt über die **Docker Engine** verwaltet werden.
* **Architektur-Muster:** Es entspricht der *Best Practice*, grundsätzlich nur **eine Applikation pro Container** auszuführen. Die Stärke von Docker liegt in der Verteilung und Entkopplung von Services.

### Komponenten einer Docker-Umgebung
* **Docker Image:** Die Applikationsumgebung inklusive aller Abhängigkeiten. Es dient als unveränderlicher Bauplan (Blueprint) für Container und besteht aus verschiedenen, wiederverwendbaren Layern.
* **Docker Container:** Die aus einem Image erzeugte, tatsächlich laufende Instanz. Nur der oberste Layer eines Containers besitzt Schreibrechte (Read-Write / RW).
* **Docker Registry:** Öffentliche oder private Repositories (z. B. Docker Hub, GitHub Container Registry `ghcr`), die zum Speichern und Verteilen von Images dienen.
* **Dockerfile:** Eine Textdatei mit sequenziellen Anweisungen zum Bauen eines Docker Images.

### CLI-Befehlsstruktur
Docker-Befehle folgen dem standardisierten Muster:
`docker <Für was> <Was tun> <Flags> <Image, Name, ID>`

*Beispiel:* `docker container run --name web-server -p 8080:80 nginx`

## 2. Architektur & Dockerfile im Detail

### Container Engine vs. Container Runtime
* **Container Engine (docker):** Verarbeitet Benutzereingaben, kommuniziert mit der API, lädt Images von Registries herunter, entpackt sie auf der Disk und bereitet Metadaten sowie Mount Points vor.
* **Container Runtime (runc):** Unterliegt dem OCI-Standard (Open Container Initiative). Sie nutzt Container Mount Points und Metadaten, kommuniziert direkt mit dem Linux-Kernel, richtet Isolation/Ressourcenregeln ein und startet den *containerized* Prozess.

### Struktur eines Dockerfiles
Ein Dockerfile wird logisch in drei Bereiche unterteilt (Ausführung von oben nach unten):
1. **Fundamental Instructions:** `FROM` (Basis-Image definieren), `ARG` (Build-Variablen).
2. **Configuration Instructions:** `RUN` (Software installieren), `ADD` / `COPY` (Dateien übertragen), `ENV` (Umgebungsvariablen setzen), `WORKDIR` (relatives Arbeitsverzeichnis festlegen).
3. **Execution Instructions:** `CMD` (Standardbefehl beim Start, überschreibbar), `ENTRYPOINT` (Fixer Startbefehl, nicht direkt überschreibbar), `EXPOSE` (Dokumentation freigegebener Ports).

> [!WARNING]
> **Layer-Caching-Problematik:** Befehle wie `apt-get update` und `apt-get install` müssen immer in einer einzigen `RUN`-Instruktion verknüpft werden (z. B. `RUN apt-get update && apt-get install -y curl`). Trennt man sie auf, wird der Update-Layer gecached. Bei späteren Rebuilds führt dies zu veralteten Paketlisten oder Installationsfehlern.

## 3. Container-Isolation & Netzwerke

### Isolation
Container sind standardmässig in Bezug auf **Prozesse, Netzwerk und Mounting** vom Host und von anderen Containern isoliert. 

### Docker-Netzwerktreiber
* **Default bridge:** Standard-Netzwerk. Bietet **keine automatische Namensauflösung (Service Discovery)**. Container können einander nur über die IP-Adresse anpringen. Der Verkehr nach aussen wird per NAT über das `docker0`-Interface des Hosts geregelt.
* **User-defined bridge:** Selbst erstellte Bridges (z. B. `docker network create --driver bridge meinebridge`) isolieren Container-Gruppen gezielt. **Wichtig:** Innerhalb eines benutzerdefinierten Bridge-Netzwerks ist die automatische Namensauflösung via DNS aktiv (Kommunikation über Container-Namen möglich).
* **host:** Bindet den Container direkt an das Netzwerkinterface des Hosts (kein virtuelles `veth`-Interface, Betrieb auf dem physischen Port des Hosts).

## 4. Docker-Storage (Datenspeicherung)
Da das Dateisystem eines Containers temporär ist und beim Löschen des Containers zerstört wird, bietet Docker drei persistente/Spezial-Optionen:
* **Volume:** Wird vollständig von Docker im Host-Pfad `/var/lib/docker/volumes/` verwaltet. Nur Docker-Prozesse (root/docker-Gruppe) haben Zugriff.
* **bind Mount:** Bindet einen beliebigen, bestehenden Ordner des Host-Dateisystems in den Container ein. Ideal für die Entwicklung, da Änderungen auf dem Host sofort im Container wirksam werden.
* **tmpfs Mount:** Schreibt Daten direkt in den flüchtigen Arbeitsspeicher (Memory) des Hosts. Bietet extrem schnelle Schreiboperationen und wird nach dem Beenden des Containers restlos abgeräumt (gut für sensitive Daten).

## 5. Multi-Container-Anwendungen mit Docker Compose
Docker Compose ermöglicht die Definition und den Start komplexer Multi-Container-Anwendungen (Microservices-Architektur) über eine einzige Konfigurationsdatei im YAML-Format (`docker-compose.yaml`).

* **Wichtigste Keywords:** `services`, `image` / `build`, `container_name`, `ports`, `networks`, `volumes`, `environment`, `depends_on` (steuert die Startreihenfolge, z. B. dass das Backend wartet, bis die Datenbank läuft).
* **Befehle:** `docker compose up -d` (Start im Hintergrund), `docker compose down` (Stoppen und Löschen aller Ressourcen).

---

## 6. Wichtigste Key Takeaways
* **Effizienz & Speed:** Container teilen sich den Host-Kernel, starten in Sekundenbruchteilen und verbrauchen drastisch weniger Ressourcen als klassische VMs.
* **Default-Einschränkung:** Das standardmässige `bridge`-Netzwerk unterstützt keine DNS-Namensauflösung zwischen Containern. Für produktive Setups müssen zwingend **User-defined Bridges** genutzt werden.
* **Persistenz-Zwang:** Daten in Containern sind flüchtig. Sollen Daten (wie Datenbanken) Updates überleben, müssen sie via **Docker Volumes** oder **bind Mounts** auf dem Host ausgelagert werden.
* **Bauen mit Verstand:** Die Reihenfolge und Struktur im Dockerfile entscheidet über die Build-Geschwindigkeit (Caching) und verhindert Sicherheitsrisiken (wie veraltete Software durch unvollständige Update-Layer).
