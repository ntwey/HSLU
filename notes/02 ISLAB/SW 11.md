## 1. Sicherheitsrisiken & Bedrohungsvektoren

### Privilegien & Root-Zugriff
* Der Docker-Daemon läuft standardmässig mit **root-Rechten**. 
* Benutzer, die zur Systemgruppe `docker` hinzugefügt werden, besitzen **indirekt vollständige root-Rechte** auf dem Host-System, da sie uneingeschränkten Zugriff auf den Daemon haben.

### Bedrohungsakteure
* **External Attacker:** Nutzt Schwachstellen in exponierten Container-Anwendungen aus (z. B. Remote Code Execution).
* **Malicious Insider:** Lokale Benutzer mit Zugriff auf die Docker-CLI, die versuchen, ihre Privilegien zu eskalieren.
* **Compromised / Poisoned Containers:** Blindes Vertrauen in Drittquellen auf Docker Hub birgt das Risiko bösartiger Images (z. B. mit integrierten Backdoors oder Krypto-Minern wie Monero).

## 2. Praktische Angreifer-Szenarien (Hacking)

### 2.1 Ausnutzen von Image-Schwachstellen (z. B. Shellshock)
* Altes oder ungepatchtes Basis-Softwarematerial in Images ermöglicht Angriffe. Bei *Shellshock* (Bash-Schwachstelle) können Angreifer über manipulierte HTTP-Header (z. B. `User-Agent`) Befehle einschleusen und eine **Reverse Shell** initialisieren, bei der das Opfersystem die Verbindung zum Angreifer aktiv aufbaut.

### 2.2 Local Privilege Escalation & Volume Mounting
* Wenn ein Benutzer der `docker`-Gruppe ein sensibles Host-Verzeichnis (oder das gesamte Root-Dateisystem `/`) via Bind-Mount in einen Container einbindet, spiegeln sich die root-Rechte des Containers im Mount-Ordner wider.
* Durch das Kompilieren einer C-Anwendung mit `setuid(0)` (Ausführung im root-Kontext) und Ablage im gemounteten Host-Ordner kann ein unprivilegierter Host-User augenblicklich echte root-Rechte auf dem Host erlangen.

### 2.3 Container Escape via Docker Socket
* Wird der Unix-Socket des Daemons (`/var/run/docker.sock`) in einen Container gemountet, erhält dieser Container die volle Kontrolle über die Docker Engine des Hosts.
* Ein Angreifer im Container kann einen neuen, privilegierten "Geschwister-Container" starten, das Root-Dateisystem des Hosts darin einhängen und so vollständig aus der Container-Isolation ausbrechen.

### 2.4 Kernelspace-Manipulation (Privileged-Flag)
* Das Starten eines Containers mit `--privileged` hebelt fast alle Isolationsmechanismen aus. Der Container erhält gefährliche Kernel-Rechte (**Capabilities**), insbesondere `CAP_SYS_MODULE`.
* Damit kann der Angreifer aus dem Container heraus ein eigens kompiliertes Linux-Kernel-Modul (`.ko`) direkt in den Kernel des Host-Systems laden (`insmod`) und besitzt damit uneingeschränkten Zugriff auf die Hardwareebene.

### 2.5 Sensitive Daten in Dangling Volumes
* Das Löschen eines Containers löscht angehängte Volumes standardmässig *nicht*. Sie verbleiben als verwaiste (**dangling**) Volumes auf der Festplatte. 
* Wenn unprivilegierte User oder bösartige Container diese via Brute-Force oder Scanning aufspüren und einbinden, können vertrauliche Restdaten (z. B. Datenbank-Credentials) gestohlen werden.

## 3. Hardening & Schutzmassnahmen

### 3.1 Linux Auditing Subsystem (auditd)
* Protokolliert Systemvorgänge auf Kernelebene. Durch das Einrichten spezifischer Überwachungsregeln (`/etc/audit/rules.d/docker.rules`) für Docker-Dateien (wie `daemon.json`, `docker.service`, `/var/lib/docker`) können administrative Manipulationen lückenlos überwacht und mittels SIEM-Visualisierungstools ausgewertet werden.

### 3.2 Sichere Docker Daemon Konfiguration (`daemon.json`)
Die Sicherheit des Daemons lässt sich über zentrale Parameter härten:
* `"icc": false`: Verbietet die unverschlüsselte Inter-Container-Kommunikation im Standardnetzwerk (`docker0`), um Sniffing-Angriffe zu blockieren.
* `"no-new-privileges": true`: Verhindert blockübergreifend, dass Prozesse über das SUID-Bit (s-Bit) neue oder höhere Rechte erlangen.
* `"userland-proxy": false`: Schaltet den langsameren Docker-Proxy ab und erzwingt das sicherere, performantere **Hairpin NAT** via iptables.
* `"live-restore": true`: Hält Container am Laufen, selbst wenn der Docker-Daemon abstürzt oder gewartet wird.

### 3.3 Das Least-Privilege-Prinzip im Container
* **Nicht als Root ausführen:** Im Dockerfile sollte zwingend ein unprivilegierter Systemnutzer angelegt und mittels `USER <name>` definiert werden.
* **Capabilities einschränken:** Docker erlaubt feingranulares Whitelisting von Kernel-Rechten. Best Practice ist das radikale Verwerfen aller Rechte und gezieltes Hinzufügen des absoluten Minimums:
  `docker run --cap-drop all --cap-add chown alpine`

### 3.4 Seccomp (Secure Computing Mode)
* Wirkt wie eine Anwendungs-Firewall für Systemaufrufe (**syscalls**). Docker nutzt standardmässig ein JSON-Profil, um gefährliche Systemaufrufe (wie `unshare`, um Namespaces zu manipulieren) abzuriegeln. Profile können kopiert und manipuliert werden, um z. B. gezielt die Rechte für Rechteänderungen (`chmod`, `fchmod`) zu entziehen.

### 3.5 AppArmor / SELinux
* Mandatory Access Control (MAC) Frameworks auf Host-Ebene. Sie erlauben das Schreiben strenger Pfad- und Operations-Richtlinien (z. B. explizites Blockieren von Lese- und Schreibrechten auf `/etc/passwd` oder `/tmp/**` für den Container), selbst wenn der Prozess im Container theoretisch root-Rechte aufweist.

### 3.6 User Namespaces (`userns-remap`)
* Abstrahiert die Benutzer-IDs. Der Root-User (UID 0) im Container wird über eine Mapping-Tabelle (`/etc/subuid`) auf eine völlig unprivilegierte, hohe UID auf dem Host (z. B. UID 231072) umgebogen. Bricht der Prozess aus, besitzt er auf dem Host keinerlei Rechte.

### 3.7 Docker Content Trust (DCT)
* Erzwingt die kryptografische Verifizierung von Images beim Download (Pull) und das Signieren beim Upload (Push) über einen Notary-Server. Verhindert das ungesehene Ausführen manipulierter oder korrumpierter Fremd-Images.
* Aktivierung via Umgebungsvariable: `export DOCKER_CONTENT_TRUST=1`

---

## 4. Wichtigste Key Takeaways
* **Root-Gefahr im Standard:** Container laufen standardmässig als `root`, und die `docker`-Gruppe ist äquivalent zu root auf dem Host. Das **Least-Privilege-Prinzip** (Zuweisung eines dedizierten `USER` im Dockerfile) ist die wichtigste fundamentale Schutzmassnahme.
* **Kein blindes Socket-Sharing:** Das Mounten des Docker-Sockets (`docker.sock`) oder das Nutzen des `--privileged`-Flags hebelt die Container-Isolation komplett aus und führt unweigerlich zum **Container Escape** (Host-Übernahme).
* **Defense in Depth:** Ein sicheres Container-Setup stützt sich nie auf nur einen Mechanismus, sondern kombiniert **Ressourcenlimits (cgroups)**, **Systemaufruf-Filterung (Seccomp)**, **Pfad-Policys (AppArmor)** und **ID-Mapping (User Namespaces)**.
* **Datenhygiene:** Beim Aufräumen von Infrastrukturen via `docker system prune` müssen zwingend auch verwaiste Volumes gelöscht werden (`--volumes`), um das Abfließen sensitiver Restdaten zu unterbinden.
