---

## 💥 Was sind Vulnerabilities?
- **Bug**: Fehler im System (kann harmlos sein)
- **Vulnerability**: Fehler mit **Sicherheitsauswirkung**, durch Angreifer ausnutzbar (z. B. CVE-2017-5754)
- Voraussetzung für Exploitierung: **Exploit** vorhanden und bekannt

### Ursachen
- Veraltete Software
- Fehlkonfigurationen
- Design-/Programmierfehler
- Menschliche Faktoren

---

## 🧠 Begriffsabgrenzung
| Begriff      | Bedeutung                                        |     |
| ------------ | ------------------------------------------------ | --- |
| **Weakness** | Technisches/organisatorisches Defizit            |     |
| **Exploit**  | Technische Umsetzung zur Ausnutzung              |     |
| **Zero-Day** | Noch ungepatchte und öffentlich unbekannte Lücke |     |

---

## 🧬 Typen von Vulnerabilities
- **Software**: SQL-Injection, Buffer Overflow, XSS, Directory Traversal, etc.
- **Menschlich**: schwache Passwörter, Phishing
- **Systemisch**: API-Fehlkonfiguration, unverschlüsselte Übertragung
- **Network/Hardware**: schlecht gesicherte Schnittstellen

---

## 🧮 Lebenszyklus einer Vulnerability
1. **Erzeugung**: Fehler entsteht
2. **Entdeckung**: intern/extern
3. **Disclosure**: Veröffentlichung
4. **Exploit verfügbar**
5. **Patch released**
6. **Patch installiert**

> Je später der Patch, desto länger das Risiko (Zero-Day → N-Day → Forever-Day)

---

## 📈 Priorisierung: CVSS
**Common Vulnerability Scoring System**

- Bewertet: **Exploitability** und **Impact**
- Skala von **0 (None)** bis **10 (Critical)**
- Beispielskriterien:
  - Angriffsvektor (AV): Network / Physical
  - Komplexität (AC)
  - Privilegien (PR)
  - Auswirkungen auf Vertraulichkeit / Integrität / Verfügbarkeit

🔗 [CVSS Calculator](https://www.first.org/cvss/calculator/3.1)

---

## 🔮 EPSS & SSVC
### EPSS – Exploit Prediction Scoring System
- Gibt Wahrscheinlichkeiten, ob eine Schwachstelle **innerhalb von 30 Tagen** ausgenutzt wird

🔗 [EPSS Model](https://www.first.org/epss/model)

### SSVC – Stakeholder-Specific Vulnerability Categorization
- Fokus auf Entscheidungshilfe für Unternehmen bzgl. Priorisierung

## 📢 Disclosure & Kommunikation
### Offenlegungsmodelle
- **Coordinated/Responsible Disclosure (CVD)**
  - Hersteller wird zuerst informiert
  - Veröffentlichung nach „angemessener Frist“

### Problemfälle
- VW vs. Forscher (Remote Keyless Entry): Veröffentlichung verboten

### ISO-Standards
- **ISO 29147** – Disclosure-Prozess
- **ISO 30111** – Vulnerability Handling

---

## 🧾 security.txt
- Standardformat (RFC 9116) zur **Kontaktaufnahme bei Sicherheitsproblemen**
- Beispiel:  
  - `https://google.com/.well-known/security.txt`  
  - `https://sbb.ch/.well-known/security.txt`

## 🧭 Zentrale Takeaways
1. **Nicht jede Schwachstelle ist ein Risiko – aber viele sind!**
2. **Priorisierung ist essenziell – nicht alles lässt sich patchen**
3. **Kommunikation (auch rechtlich/ethisch) ist zentral**
4. **Security ist auch Verantwortung!**
